# WCAG 2.1 / 2.2 AA コードレビュー チェックリスト

フロントエンドコード（HTML/JSX/TSX/Vue）のレビューで頻出する達成基準を、コードレベルの確認内容に翻訳したもの。DADS が目標とする **適合レベル AA**（A + AA）を対象とし、参考として AAA・WCAG 2.2 追加基準も含める。

各項目の見出しは `達成基準番号 名称 (レベル)`。コードから断定できない項目は「要・実機確認」と記す。

## 1. 知覚可能 (Perceivable)

### 1.1.1 非テキストコンテンツ (A)
- `img` に `alt` があるか。意味を持つ画像は内容を表す alt、装飾画像は `alt=""`（または CSS 背景画像）。
- アイコンのみのボタン/リンクに `aria-label` か視覚的に隠したテキストがあるか。`alt="image"` のような無意味な値になっていないか。
- `svg` に `role="img"` + `aria-label`、または `<title>`。純装飾の svg は `aria-hidden="true"`。
- `input type="image"` に `alt`。CAPTCHA・グラフ・図表に代替手段があるか。

### 1.2.x 時間依存メディア (A/AA)
- `video`/`audio` に字幕（`track kind="captions"`）、音声解説、書き起こしがあるか。要・実機確認の場合が多い。

### 1.3.1 情報及び関係性 (A)
- 見出しは `h1`〜`h6` でマークアップされているか。見た目だけ大きい `div` で代用していないか。
- 見出しレベルが飛んでいないか（`h2` の次に `h4` など）。ページに `h1` があるか。
- リストは `ul`/`ol`/`li`、定義リストは `dl`。視覚的なリストを `div` の羅列にしていないか。
- データテーブルは `table` + `th`（`scope="col"`/`scope="row"`）。レイアウト目的に table を使っていないか。`td` を見出しに誤用していないか。
- フォーム入力に `label`（`for`/`id`）が関連付いているか。グループは `fieldset`+`legend`。
- ランドマーク（`header`/`nav`/`main`/`footer` または `role`）で領域が区別されているか。
- 太字・色・位置といった視覚表現だけで伝えている関係を、マークアップでも表現しているか。

### 1.3.2 意味のある順序 (A)
- DOM 順序が読み上げ順序として意味が通るか。CSS（`order`, `position`, `float`, `flex-direction: row-reverse`）で視覚順序と DOM 順序が乖離していないか。

### 1.3.3 感覚的な特徴 (A)
- 「右のボタン」「丸いアイコン」など、形・位置・色だけに依存した説明になっていないか。

### 1.3.4 表示の向き (AA)
- `orientation` を縦/横どちらかに固定していないか（CSS の向きロックなど）。

### 1.3.5 入力目的の特定 (AA)
- 氏名・メール・電話・住所などの入力に適切な `autocomplete` 属性があるか。

### 1.4.1 色の使用 (A)
- 色だけで情報・状態・操作を伝えていないか（必須項目を赤字だけ、エラーを赤枠だけ、リンクを色だけで区別など）。アイコン・テキスト・下線などを併用しているか。

### 1.4.3 コントラスト (最低限) (AA)
- テキストと背景のコントラスト比が **通常 4.5:1 以上**、**大きい文字（18pt/24px、または太字 14pt/18.66px 以上）は 3:1 以上**か。
- コード上に前景色・背景色の値があれば比を計算する。`opacity` 半透明、画像の上のテキスト、グラデーション、ホバー/プレースホルダの淡色は見落としやすい。値が確定しない場合は **要・実機確認**。

### 1.4.4 テキストのサイズ変更 (AA)
- 200% 拡大で破綻しないか。フォントサイズに `px` 固定でズーム不可になっていないか（`rem`/`em` 推奨）。`user-scalable=no`, `maximum-scale=1` を viewport に設定していないか。

### 1.4.5 文字画像 (AA)
- ロゴ以外でテキストを画像化していないか。

### 1.4.10 リフロー (AA)
- 幅 320px 相当で横スクロールが発生しないか。固定幅レイアウト、`white-space: nowrap` の多用、`overflow` 切れに注意。

### 1.4.11 非テキストのコントラスト (AA)
- UI コンポーネントの境界（入力枠、ボタン、フォーカスリング）やアイコン、グラフの要素が背景に対し **3:1 以上**か。

### 1.4.12 テキストの間隔 (AA)
- 行間・段落間・文字間をユーザーが広げても内容が切れないか。`height` 固定 + `overflow: hidden` のテキストコンテナに注意。

### 1.4.13 ホバー又はフォーカスで表示されるコンテンツ (AA)
- ツールチップ/ポップオーバーが、消去可能・ホバー維持可能・ポインタを外すまで表示維持か。

## 2. 操作可能 (Operable)

### 2.1.1 キーボード (A)
- すべての操作がキーボードで可能か。`div`/`span` に `onClick` だけ付けてキーボード非対応になっていないか（→ `button` を使うか、`role` + `tabindex="0"` + `onKeyDown` で Enter/Space に対応）。
- カスタムドロップダウン、タブ、アコーディオン、スライダーが矢印キー等で操作できるか。

### 2.1.2 キーボードトラップなし (A)
- モーダル/ウィジェットからキーボードで抜け出せるか。フォーカスが閉じ込められて戻れない実装になっていないか。

### 2.1.4 文字キーのショートカット (A)
- 単一文字のショートカットを、無効化・再マッピング・フォーカス時のみ有効のいずれかにできるか。

### 2.4.1 ブロックスキップ (A)
- 「メインコンテンツへスキップ」リンク、または適切なランドマークで繰り返しブロックを飛ばせるか。

### 2.4.2 ページタイトル (A)
- `title`（または SPA のルートごとの `document.title`）が内容を表すか。

### 2.4.3 フォーカス順序 (A)
- Tab 順序が意味の通る順か。`tabindex` に正の値（`tabindex="1"` 以上）を使っていないか（順序破壊の原因）。

### 2.4.4 リンクの目的（コンテキスト内） (A)
- リンクテキストだけ、または周辺コンテキストで目的が分かるか。「こちら」「詳細」「もっと見る」の多用に注意。アイコンのみのリンクにアクセシブルな名前があるか。

### 2.4.5 複数の手段 (AA)
- サイト内でページに到達する手段が複数あるか（ナビ、検索、サイトマップ等）。

### 2.4.6 見出し及びラベル (AA)
- 見出し・ラベルが内容や目的を的確に表しているか。

### 2.4.7 フォーカスの可視化 (AA)
- フォーカスインジケータが見えるか。`outline: none` / `outline: 0` でフォーカスリングを消して代替を用意していない実装に注意（**must 級の頻出**）。

### 2.5.1 ポインタのジェスチャ (A)
- ピンチ・スワイプ等の複雑なジェスチャに、単純なポインタ操作の代替があるか。

### 2.5.2 ポインタのキャンセル (A)
- `mousedown`/`pointerdown` で確定実行せず、`click`（up イベント）で実行しているか。誤操作を取り消せるか。

### 2.5.3 名前（ラベル）に含まれる文字列 (A)
- 視覚ラベルのテキストが、アクセシブルな名前（`aria-label` 等）に含まれているか。音声操作で食い違わないか。

### 2.5.4 動きによる起動 (A)
- 端末を振る等の動きで起動する機能に、UI 操作の代替と無効化手段があるか。

## 3. 理解可能 (Understandable)

### 3.1.1 ページの言語 (A)
- `html` に `lang`（日本語なら `lang="ja"`）があるか。

### 3.1.2 一部分の言語 (AA)
- 本文と異なる言語の箇所に `lang` 指定があるか。

### 3.2.1 フォーカス時 (A) / 3.2.2 入力時 (A)
- フォーカスや入力だけで、文脈の予期しない変化（自動送信、ページ遷移、新規ウィンドウ）が起きないか。`onChange` での自動 submit に注意。

### 3.2.3 一貫したナビゲーション (AA) / 3.2.4 一貫した識別性 (AA)
- 共通ナビの順序が一貫しているか。同じ機能のコンポーネントに一貫したラベル/アイコンを使っているか。

### 3.3.1 エラーの特定 (A)
- 入力エラーが箇所とともにテキストで示されるか。色だけで示していないか。

### 3.3.2 ラベル又は説明 (A)
- 入力に可視ラベルや説明があるか。プレースホルダのみをラベル代わりにしていないか（フォーカスで消えて参照不能になる）。
- 必須項目が `required`/`aria-required` と視覚表示の両方で示されているか。

### 3.3.3 エラー修正の提案 (AA)
- エラー時に修正方法の提案があるか（形式の例示など）。

### 3.3.4 エラー回避（法的、金融、データ） (AA)
- 送信前に確認・修正・取り消しができるか。

## 4. 堅牢 (Robust)

### 4.1.2 名前・役割・値 (A)
- カスタムコンポーネントに適切な `role`、アクセシブルな名前（`aria-label`/`aria-labelledby`）、状態（`aria-expanded`/`aria-checked`/`aria-selected`/`aria-disabled`）があるか。
- ネイティブ要素（`button`/`a`/`input`/`select`）で済むものを `div` + ARIA で再実装していないか（**ネイティブ優先**）。
- `aria-*` 属性名・値が正しいか。`role` に対し必須の属性が揃っているか。

### 4.1.3 ステータスメッセージ (AA)
- 動的な通知（保存完了、検索件数、エラー件数）が `aria-live`（`role="status"`/`role="alert"`）でフォーカス移動なしに伝わるか。

### （4.1.1 構文解析 は WCAG 2.2 で廃止）
- 重複 `id`、属性の不正など。2.2 では達成基準から除外されたが、`id` 重複は `label`/`aria-*` の参照を壊すため引き続き確認する。

## 5. WCAG 2.2 追加基準

DADS は WCAG 2.2 の A・AA も順次対象に加える方針。新規実装ではこれらも確認する。

### 2.4.11 フォーカスの非遮蔽（最小限） (AA)
- フォーカスされた要素が、固定ヘッダー/フッターやスティッキー要素に完全には隠されないか。

### 2.5.7 ドラッグ動作 (AA)
- ドラッグ&ドロップ操作に、単一ポインタ（クリック等）の代替があるか。並べ替え UI に注意。

### 2.5.8 ターゲットのサイズ（最小限） (AA)
- クリック/タップ対象が **24×24 CSS px 以上**、または十分な間隔があるか。アイコンボタン、近接したリンクに注意。

### 3.2.6 一貫したヘルプ (A)
- ヘルプ手段（問い合わせ先等）が複数ページで一貫した位置にあるか。

### 3.3.7 冗長な入力 (A)
- 同一プロセス内で既出の情報を再入力させていないか（自動入力・選択肢提示）。

### 3.3.8 アクセシブルな認証（最小限） (AA)
- ログイン等で、記憶・パズル・文字起こしを必須にしていないか（パスワード貼り付け禁止、認知テスト必須などに注意）。

## レビュー時の補足

- **ネイティブ要素優先**: 多くの 4.1.2 / 2.1.1 違反は、`div` で作り直さずネイティブ要素を使えば解決する。代替の提案では常にこれを第一候補にする。
- **ARIA の濫用に注意**: 不要・誤った ARIA はないより悪い。`role="button"` を `button` に付ける、`aria-label` で可視テキストと食い違う等。
- **動的 UI**: モーダル・ドロワー・トースト・タブ・コンボボックスは、フォーカス管理（開いたら中へ、閉じたら呼び出し元へ）、`aria-expanded`/`aria-modal`、Esc で閉じる、を一式で確認する。
- **コードで断定できないもの**: コントラスト比の実測、読み上げ順・読み上げ内容、フォーカスリングの視認性、拡大時の破綻は、レポートの「要・実機確認」に回し、axe DevTools / Lighthouse / スクリーンリーダー（NVDA・VoiceOver）での確認を促す。
