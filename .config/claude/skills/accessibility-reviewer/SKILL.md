# アクセシビリティレビュー (DADS / WCAG 2.1・2.2 AA 準拠)

フロントエンドのコード（HTML / JSX / TSX / Vue / テンプレート）を、アクセシビリティの観点でレビューする。

基準はデジタル庁デザインシステム（DADS）の方針に合わせる。DADS は **JIS X 8341-3:2016** と **WCAG 2.1 / 2.2** に基づき、**適合レベル AA** を目標としている。したがって本スキルも A・AA の達成基準を対象とし、AAA は参考扱いとする。

## レビューの進め方

1. **対象を確定する。** レビュー対象のファイル・コンポーネント・差分を特定する。範囲が曖昧なら、変更差分（`git diff`）か指定ファイルに絞る。広すぎると指摘が薄くなるので、画面・コンポーネント単位に区切る。

2. **コードを読み、UI の意味構造を把握する。** 何の要素か（ボタン/リンク/入力/見出し/画像/ランドマーク）、状態（開閉・選択・無効・エラー）、操作（クリック/キーボード/フォーカス移動）を読み取る。動的に組み立てられる UI（条件レンダリング、リスト、モーダル）は、生成される DOM を想像して評価する。

3. **チェックリストで照合する。** `references/wcag-checklist.md` を読み、関連する達成基準と照合する。全項目を機械的に並べるのではなく、対象コードに**実際に関係する基準**だけを取り上げる。チェックリストはコードレビューで頻出する基準に絞ってあるので、まずこれを基準にする。

4. **指摘を分類して報告する。** 後述の出力形式に従う。各指摘には必ず**達成基準番号と適合レベル**を添える（例: `1.1.1 非テキストコンテンツ (A)`）。なぜ問題か（どの利用者がどう困るか）と、具体的な修正コードを示す。

5. **推測と事実を区別する。** コードだけでは判定できない基準（色のコントラスト比の実測値、実際の読み上げ順、フォーカスの見え方など）は「コードからは断定できない。要・実機確認」と明記する。憶測で must を出さない。

## 出力形式

ユーザーの review-mode 規約に従い、指摘を 4 段階に分類する。

- **must**: 達成基準 A・AA への不適合。修正必須。
- **imo**: AA は満たすが改善が望ましい点、AAA 相当の推奨。
- **nits**: 軽微なマークアップ・命名の指摘。
- **q**: 意図の確認（動的挙動や設計意図が読み取れないとき）。

以下のテンプレートで日本語の Markdown レポートを出力する。

```markdown
# アクセシビリティレビュー結果

対象: <ファイル/コンポーネント/差分>
基準: WCAG 2.1/2.2 AA + JIS X 8341-3:2016（DADS 準拠）

## サマリ
- must: N件 / imo: N件 / nits: N件
- 総評: <1〜2文。最も重要な不適合は何か>

## 指摘

### [must] <見出し> — `1.3.1 情報及び関係性 (A)`
**該当**: `path/to/file.tsx:42`
**問題**: <どの利用者がどう困るか>
**修正案**:
​```tsx
// before / after
​```

### [imo] ...

## 要・実機確認
- <コントラスト比の実測、読み上げ順、フォーカスリングの視認性など、コードだけでは判定できない項目>
```

指摘が一件もない場合も、何を確認した結果問題がなかったのかを「確認済みの観点」として簡潔に列挙し、空のレポートにしない。

## 重点的に見る観点（DADS の不適合事例より）

DADS 自身の検証で実際に不適合となった項目は、現場でも頻出する。優先的に確認する。

- **見出し構造**: `h1` の有無、見出しレベルの飛び（`h2` の次に `h4` など）。`1.3.1`
- **画像・アイコンの代替**: 意味を持つ画像の `alt`、装飾画像の `alt=""`、アイコンのみのボタンのラベル。`1.1.1`
- **フォームのラベル**: `label` と入力の関連付け（`for`/`id` または `aria-label`）。プレースホルダだけで代用しない。`1.3.1` `3.3.2` `4.1.2`
- **ボタン・リンクのテキスト**: 「こちら」「詳細」だけのリンク、テキストのないアイコンボタン。`2.4.4` `2.4.6` `4.1.2`
- **キーボード操作**: `div`/`span` に `onClick` だけ付けてキーボード操作不可になっていないか。ドロップダウン・モーダルの操作とフォーカス管理。`2.1.1` `2.1.2`
- **テーブル**: データテーブルでの `th`/`scope`、`td` の誤用。`1.3.1`
- **色のコントラスト**: テキストと背景の比（通常 4.5:1、大きい文字 3:1）。コード上のカラー値から比を計算できる場合は計算し、できない場合は要確認とする。`1.4.3`
- **色だけで情報を伝えていないか**: エラーを赤色だけで示す等。`1.4.1`

## 詳細チェックリスト

達成基準ごとのコードレベルの確認内容は `references/wcag-checklist.md` にまとめてある。レビュー時はこれを読んで照合する。WCAG 2.2 で追加された基準（フォーカスの非遮蔽、ターゲットサイズ、ドラッグ操作の代替など）も含む。
