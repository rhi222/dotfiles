# esa 週次エグゼクティブレポート生成

複数のesa記事の直近一週間の差分を収集し、部長会向けエグゼクティブレポートを生成する。

追加の記事指定（任意）: $ARGUMENTS
形式: `<team> <post_number> [<team> <post_number> ...]`

## 参照スキル

- `.config/claude/skills/esa-api/SKILL.md` — esa API アクセス方法

## 手順

### 1. 対象記事リストの構築

1. `.config/claude/skills/esa-weekly-report/esa-weekly-report-posts.json` から固定リストを読み込む
2. `$ARGUMENTS` に追加の `<team> <post_number>` ペアがあればリストに追加

### 2. 各記事の差分収集

対象記事ごとに以下を実行する。各記事は独立しているため並列Bash呼び出しで処理すること。

**差分の集計期間は「定例と定例の間」とする。** 定例は毎週金曜 14:00 に開催され、その場で表示される内容は「金曜 14:00 直前時点のリビジョン」である。したがって収集したいのは `前回の金曜 14:00 定例時点 → 今回の金曜 14:00 定例時点` の差分。

1. `esa-api` スキルに従い、リビジョン一覧を取得
2. 前回定例の日時を特定: 今回の定例（直近の金曜 14:00）の1つ前の金曜 14:00（= 前回SoSで参照された時点）
   - 例: 今回の定例が 07-17(金) なら、前回定例は 07-10(金) 14:00
3. FROM を決定: **前回の金曜 14:00 より前（strictly before）に作成された最新リビジョン**（= 前回定例で表示されていた内容）
   - `created_at < 前回金曜 14:00` を満たすリビジョンのうち最新のもの
   - 注意: 金曜 14:00 **以降**（定例中〜定例後）に編集されたリビジョンは前回定例には表示されていないため、今回の差分に含める。この判定を誤ると FROM が1つ後ろにずれる（例: 検索 rev140=金14:27 を選ぶと誤り、正しくは rev139=金13:59）
4. レポートに載せる compare URL を生成: `https://<team>.esa.io/posts/<number>/revisions/compare/<from>...head/html_diff`
   - ブラウザURLは TO を `head` とする。SoS直前に記事が追記更新された場合でもURLを書き換えずに最新差分を表示できる
5. compare API で diff 内容を取得
   - API は `head` キーワード非対応（404）。API呼び出しでは具体的なリビジョン番号（現時点の最新番号を TO）を指定する
   - diff は `diff_text` フィールド（`.diff_text // .diff.diff_text`）が読みやすい。`+`/`-` 行を抽出して使う

**差分が無い場合:**
- 前回金曜 14:00 以降にリビジョンが増えていない → 「変更なし」と記録し、レポートでは省略

### 3. エグゼクティブレポート生成

収集した全記事の差分を統合し、以下の構造でレポートを作成する:

```markdown
# 週次レポート（YYYY-MM-DD）

## サマリ
- 【検索】変更概要（1行）
- 【素材登録】変更概要（1行）
- 【TACO】変更概要（1行）
- 【予約】変更概要（1行）
- 【基盤 / システム名】変更概要（1行）
- ...
※ サブシステム名は記事の label を使用
※ 基盤のみ、記事内のシステム名（航空キュー基盤、料金計算基盤 等）の粒度で分けて記載
※ リスクや特に注意が必要な事項があれば ⚠ を付けて明示

## 記事別の変更点

### [label]
- compare URL: <url>
- 主な変更点
  - ...

### [label]
...
```

レポートのルール:
- 記事別の変更点: diff から読み取れる事実のみ記載
- サマリ: 事実に基づく仮説や示唆を含めてよいが、仮説部分は「（仮説）」と明示する
- 技術的詳細は最小限に抑える
- 出力後にユーザーがレビューするため、仮説の正誤はユーザー判断に委ねる

### 4. 出力

1. **ファイル保存**: `~/Obsidian/05_Organization/Buchokai/YYYY-MM-DD.md` に保存
   - ファイル名の日付は今日の日付
   - 既存ファイルがある場合はユーザーに確認
2. **会話内表示**: レポートのサマリを会話内にも表示
